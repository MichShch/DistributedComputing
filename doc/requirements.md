# Требования к системе распределённых вычислений

## 1. Общая информация

Проект представляет собой прикладную систему управления распределёнными вычислениями, реализованную на языке C++ с поддержкой платформ Windows и Linux.  
Система предоставляет CLI-интерфейс как основной способ управления, а также API для интеграции с GUI-приложением на C#.

---

## 2. Архитектура системы

Система состоит из следующих компонентов:

- **Master (Control Service)** — центральный управляющий сервис
- **Worker Agent** — агент выполнения задач на вычислительных узлах
- **CLI-модуль** — инструмент командной строки для управления системой
- **External GUI (C#)** — вторичный клиент, использующий API Master

Взаимодействие компонентов осуществляется через REST или gRPC API.

---

## 3. Требования к Worker Agent

### 3.1 Назначение
Worker Agent предназначен для выполнения вычислительных задач путём запуска внешних программ с заданными параметрами.

---

### 3.2 Функциональные требования

#### 3.2.1 Регистрация и идентификация
Агент должен:
- регистрироваться в Master при запуске
- передавать уникальный идентификатор
- передавать информацию об ОС, версии агента и доступных ресурсах
- поддерживать механизм heartbeat

---

#### 3.2.2 Получение задач
Агент должен:
- периодически запрашивать задачи у Master
- принимать задачи только при наличии свободных слотов
- отклонять задачи, несовместимые с ОС или ресурсами

---

#### 3.2.3 Выполнение задач
Агент должен:
- запускать внешние программы как отдельные процессы ОС
- передавать аргументы командной строки
- поддерживать переменные окружения
- поддерживать тайм-аут выполнения

---

#### 3.2.4 Управление процессами
Агент должен:
- уметь принудительно завершать задачи по запросу Master
- корректно завершать дочерние процессы
- предотвращать утечки процессов

---

#### 3.2.5 Отчётность и логирование
Агент должен:
- отправлять статус задачи в Master
- сохранять stdout и stderr в лог-файлы
- логировать собственные события и ошибки

---

### 3.3 Нефункциональные требования
- кроссплатформенность (Windows / Linux)
- устойчивость к сбоям сети
- автоматическое восстановление после перезапуска

---

## 4. Требования к Master (Control Service)

### 4.1 Назначение
Master отвечает за централизованное управление задачами и агентами, а также предоставляет API для внешних клиентов.

---

### 4.2 Функциональные требования

#### 4.2.1 Управление агентами
Master должен:
- принимать регистрацию агентов
- отслеживать состояние агентов (online / offline / busy)
- удалять агентов при отсутствии heartbeat

---

#### 4.2.2 Управление задачами
Master должен:
- принимать задачи от клиентов
- хранить параметры и состояние задач
- поддерживать очередь задач

---

#### 4.2.3 Планирование задач
Master должен:
- распределять задачи с учётом ОС и доступных ресурсов
- предотвращать дублирующее выполнение задач

---

#### 4.2.4 Контроль выполнения
Master должен:
- принимать отчёты о выполнении задач
- поддерживать отмену задач
- фиксировать причины ошибок

---

#### 4.2.5 API
Master должен:
- предоставлять REST или gRPC API
- обеспечивать доступ к задачам, агентам и логам

---

### 4.3 Нефункциональные требования
- масштабируемость (десятки агентов)
- потокобезопасность
- сохранение состояния при перезапуске

---

## 5. Требования к CLI-модулю

### 5.1 Назначение
CLI-модуль предназначен для управления и мониторинга системы из командной строки.

---

### 5.2 Функциональные требования

#### 5.2.1 Управление задачами
CLI должен поддерживать:
- отправку задач
- просмотр списка задач
- получение детального статуса задачи
- отмену задачи

---

#### 5.2.2 Управление агентами
CLI должен поддерживать:
- просмотр списка агентов
- получение детальной информации об агенте

---

#### 5.2.3 Работа с логами
CLI должен:
- получать stdout/stderr задач
- поддерживать режим просмотра логов в реальном времени

---

### 5.3 Нефункциональные требования
- работа в headless-среде
- минимальные зависимости
- человекочитаемый и JSON-форматы вывода

---

## 6. Сквозные требования

### 6.1 Безопасность
- контроль допустимых программ для запуска
- защита от произвольного выполнения команд

---

### 6.2 Надёжность
- повторное назначение задач при падении агента
- устойчивость к сбоям Master

---

## 7. Критерии готовности

Система считается готовой, если:
- задачи успешно выполняются на Windows и Linux
- управление полностью доступно через CLI
- GUI может быть подключён через API без изменения логики
